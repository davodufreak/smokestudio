## Vulnerabilidades de Seguridad

### [CRÍTICA] XSS — Input del usuario inyectado en innerHTML

**Archivo:** `smokescan.js:1486`

```js
// VULNERABLE
chatWindow.innerHTML += `<div ...>${txt}</div>`;
```

El texto del chatbot que escribe el usuario (`txt`) se inserta directamente en `innerHTML` sin ninguna sanitización. Un mensaje como `<img src=x onerror=alert(document.cookie)>` ejecuta JavaScript arbitrario en el contexto del sitio.

**Impacto:** Robo de sesión, ejecución de código malicioso, phishing dentro del sitio.

**Fix requerido:**
```js
// CORRECTO
const div = document.createElement('div');
div.textContent = txt;
// aplicar estilos como propiedades de style...
chatWindow.appendChild(div);
```

---

### [ALTA] XSS — Contenido de Gemini renderizado sin sanitizar

**Archivos:** `smokescan.js:829-874`

Los campos `cat.name`, `cat.desc`, `step.title`, `step.description` de la respuesta del proxy Gemini se insertan via template literals en `innerHTML`. Si el Edge Function es comprometido o la API devuelve contenido malicioso, se ejecuta JS en el cliente.

**Fix requerido:** Usar `DOMPurify.sanitize()` o asignar con `element.textContent` para campos de texto plano.

---

### [ALTA] Bug Crítico — `getScoreStatus()` no definida

**Archivo:** `smokescan.js:1183`

```js
const newResult = getScoreStatus(); // ReferenceError: getScoreStatus is not defined
```

Se llama en `applyChangesAndReanalyze()`, que se activa cuando el usuario edita respuestas desde el panel de resultados. Cualquier edición post-análisis produce un crash (`ReferenceError`) y congela la UI.

**Fix requerido:** Reemplazar con `generateSimulatedResult()`.

---

### [ALTA] `updateBreakdown` definida dos veces

**Archivo:** `smokescan.js:132` y `smokescan.js:773`

La segunda definición (línea 773) sobrescribe silenciosamente la primera (línea 132) en runtime. La primera nunca se ejecuta. Aunque ambas son funcionalmente equivalentes, la duplicación es un indicador de deuda técnica y puede causar regresiones al modificar una sin la otra.

**Fix requerido:** Eliminar la definición de la línea 132 y dejar únicamente la de línea 773 (que tiene mejor lógica de prefix).

---

### [MEDIA] `saveAnalysis()` nunca se llama

**Archivo:** `smokescan.js:1295-1304`

La función que persiste análisis en Supabase existe pero ningún flujo la invoca. Los análisis completados **nunca se guardan en la base de datos**.

**Fix requerido:** Llamar `saveAnalysis()` al final de `showResults()`.

### [MEDIA] `SUPABASE_ANON_KEY` hardcodeada en el repositorio

**Archivo:** `smokescan.js:1213`


El anon key es público por diseño de Supabase y seguro siempre que RLS esté correctamente configurado. Sin embargo, tenerlo versionado en git dificulta la rotación de claves y expone el Project Ref públicamente.

### [MEDIA] `raw_user_meta_data` usado para nombre de usuario

**Archivo:** `smokescan.js:1282`

`user_metadata` es editable por el usuario (campo inseguro para autorización según el skill de Supabase). Aunque aquí solo se usa para mostrar el nombre, si ese nombre se renderiza vía `innerHTML` en algún punto, se convierte en vector de stored XSS.

## Backend / Supabase

### Problema 1: Parsing frágil de respuesta Gemini

**Archivo:** `smokescan.js:577-579`

El código asume que el proxy retorna el shape exacto de la API de Gemini (`candidates[0].content.parts[0].text`). Si el Edge Function wrappea la respuesta, falla silenciosamente y cae al resultado simulado sin registrar el error real.

**Recomendación:** Agregar logging del error real en el catch, y documentar el contrato del Edge Function (qué shape devuelve el proxy).

---

### Problema 2: Sin validación de tamaño de prompt en el proxy

El Edge Function `gemini-proxy` recibe `{ prompt: string }` sin límite de tamaño. Un actor malicioso puede enviar prompts masivos y generar costos elevados en Gemini API.

**Recomendación:** Agregar en el Edge Function:
```ts
if (prompt.length > 4000) {
  return new Response('Prompt too long', { status: 400 });
}
```

---

### Problema 3: RLS — verificar políticas en las 3 tablas

Según el skill `supabase-postgres-best-practices`, las tablas expuestas deben tener RLS activado. Para este proyecto:

| Tabla | SELECT | INSERT | UPDATE | DELETE |
|-------|--------|--------|--------|--------|
| `users` | `auth.uid() = id` | Solo sistema | Solo sistema | No |
| `analyses` | `auth.uid() = user_id` | `auth.uid() = user_id` | No | No |
| `leads` | Solo admin | Autenticado | No | No |
